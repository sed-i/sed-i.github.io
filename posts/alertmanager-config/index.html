    Playing with alertmanager config

    July 22, 2021
    observability
    alerts, alertmanager, exploration

    Some people get excited by dates such as April 20, 2004, 8:04pm (20/04/2004, 20:04). Well, what if we could have alertmanager fire an alert every time the gregorian calendar looks exciting?

    alertmanager does offers quite a powerful configurational flexibility that we could use just for that.

    For the purpose of this exercise we will be generating dummy alerts, based on $elapsed=$(date +%s), the number of seconds since 1970-01-01, as follows:

    service alertname label: property
    prime $(date +%s) “prime”, if the number is a prime number
    palindrome $(date +%s) “palindrome”, if the number is a palindrome
    fizzbuzz $(date +%s) “Fizz”, “Buzz” or “FizzBuzz”

    Install locally (snap) # 

    > $ sudo snap install prometheus-alertmanager

    After which you can see that alertmanager is running with default parameters:

    > $ pgrep -a alertmanager
/snap/prometheus-alertmanager/11/bin/alertmanager --config.file=/var/snap/prometheus-alertmanager/11/alertmanager.yml --storage.path=/var/snap/prometheus-alertmanager/common

    Note the default location of the configuration file: cat /var/snap/prometheus-alertmanager/11/alertmanager.yml.

    Anatomy of an alert #

    The two most important attributes of an alarm are status and labels. In yaml form it looks like this:

    status: firing
labels:
  alertname: "name of alert"
  service: "name of service"
  severety: "warning"

    Alerts like this one can be posted to the alertmanager server via HTTP API, obtained via HTTP API or viewed on a web gui via a browser.

    Just to get things started, we can easily create a simple alert with the following bash function:

    post_alert () {
    # $1 service
    # $2 alertname
    # $3 labels.property
    
    curl -XPOST $url -d "[{ 
        \"status\": \"firing\",
        \"labels\": {
                \"alertname\": \"$2\",
                \"service\": \"$1\",
                \"severity\":\"warning\",
                \"property\": \"$3\"
        },
        \"annotations\": {
                \"summary\": \"$1: $2 $3\"
        },
        \"generatorURL\": \"http://prometheus.int.example.net\"
    }]" && echo ""
}

    Alerting model #

    For the purpose of this exercise we will be generating dummy alerts, based on the number of seconds since 1970-01-01 (obtained via date +%s), as follows:

    service alertname label: property
    prime $(date +%s) “prime”, if the number is a prime number
    palindrome $(date +%s) “palindrome”, if the number is a palindrome
    fizzbuzz $(date +%s) “Fizz”, “Buzz” or “FizzBuzz”

    This way we will have many alerts (possibly duplicated) to play, group and filter with.

    alertmanager configuration # 

    > $ cat /var/snap/prometheus-alertmanager/11/alertmanager.yml
route:
  group_by: ['alertname']
  receiver: 'dummy'
receivers:
- name: 'dummy'
    Parameter Purpose
    group_by: ['alertname'] what groups divide alerts into
    group_wait: 30s collect a bunch of alerts before notifying
    group_interval: 5m wait before re-notifying (per group)
    repeat_interval: 1h wait before re-notifying (per alert)

    The alert-generating script # 

    while :
do
    elapsed=$(date +%s)
    is_prime $elapsed
    is_palindrome $elapsed
    fizzbuzz $elapsed
    echo "-------------------" 
    sleep 0.3
done

    See alerts in a karma dashboard #

    The resulting alerts are easy to work with using a karma dashboard.

    We can notice a few magical things the happened to our alerts:

    1. Even though we only sleep 0.3 and generate duplicated alerts for the same value of $elapsed, alertmanager de-duplicated those alerts without us even asking for it.
    2. The alerts are indeed grouped by alertname: all the alerts for a given value of $elapsed are grouped together. For example, 1626995313 is divisible by 3, its last 3 digits (313) form a palindrome, and its last two digits (13) is a prime number. On the other hand, 1626995330 is divisible by 5 but does not have the other properties.
    3. The service that generated the alerts is clearly visible (and can be used for filtering). If we wanted to see all the numbers that are a palindrome, we could group_by: ['service'] and then filter by service=palindrome.

    Inhibit rules #

    Let’s change the logic of the fizzbuzz function to print ‘Fizz’, ‘Buzz’ AND ‘FizzBuzz’ when the number is divisible by 15. Now we will have three alarms for every ‘FizzBuzz’.

    An inhibit rule allows to mute a set of alerts given that another alert is firing. In the FizzBuzz case, we want to mute both Fizz and Buzz if FizzBuzz is already firing:

    inhibit_rules:
  - source_match:
      service: 'fizzbuzz'
      property: 'FizzBuzz'
    target_match_re:
      service: 'fizzbuzz'
      property: 'Fizz|Buzz'
    equal: ['alertname']
